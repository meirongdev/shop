## 一、背景与目标

### 1.1 问题现状

- `docs/`（31 个工程内部文档）与 `docs-site/`（10 个 Docusaurus 页面）并行维护，存在内容漂移风险
- `docs-site/docs/services/{core,growth,platform}.md` 已拆分领域内容，按职责分组展示
- Sidebar 完全自动生成（`autogenerated`），无法体现清晰的学习路径
- 技术栈介绍仅有一张简表，缺乏选型决策与实战模式
- 无明确的"快速开始"路径

### 1.2 目标

| 优先级 | 目标 |
|--------|------|
| P0 | 展示：可以直接给他人看的高质量公开文档站 |
| P1 | Onboarding：新开发者能快速理解架构并运行项目 |
| P2 | 学习参考：技术选型决策 + 实战模式可回查 |

### 1.3 设计原则

- **docs-site 作权威展示层**：对外内容统一在 docs-site 维护，不与 docs/ 重复同步
- **docs/ 作工程工作区**：superpowers/specs、plans 保留；内部详细文档保留但不对外
- **单一认知路径**：sidebar 顺序即学习顺序

## 四、各页面内容规划

### 4.1 `intro.md` — 项目概览（重写）

- 一句话定义：云原生微服务电商技术验证平台
- 核心价值主张：验证 Gateway + Thin BFF + Domain Service 在 Java 25 / Spring Boot 3.5 下的可行性
- 服务全景图（精简版，带端口号，含 Profile Service）
- 快速导航卡片（→ 快速开始 / → 架构 / → GitHub）

### 4.2 `getting-started/quick-start.md`（新建）

**主路径：Docker Compose**（最快，无需 Kind）

- 前置条件：Docker Desktop、Java 25、Maven
- 四步跑起来：
  1. `cp .env.example .env`
  2. `./mvnw -q package -DskipTests`
  3. `docker compose up -d`
  4. 浏览器访问 `http://localhost:8080`
- Smoke check 命令（登录、搜索商品、下单各一条 `curl`）
- 常见问题（Redis Port 环境变量冲突、Kafka 启动顺序）

**进阶路径**：链接到 `local-deployment.md`（Kind + mirrord 完整说明）

### 4.3 `tech-stack/index.md`（新建，展示重点）

**范围：技术全景 + 选型决策表，正文控制在 ≤400 词**

不在此页放代码片段——代码示例统一在 `best-practices.md`。

```
技术全景图（分层可视化：Client → Gateway → BFF → Domain → Infra）
↓
选型决策表（每项技术一行：选了什么 / 为什么选 / 不选什么）
```

| 类别 | 选择 | 为什么选 | 没选什么 |
|------|------|---------|---------|
| 运行时并发 | Java 25 Virtual Threads | 同步代码可读性 + 高吞吐，无需响应式编程 | WebFlux / Reactive |
| 网关 | Spring Cloud Gateway MVC | 与 Virtual Threads 栈一致，Servlet 生态兼容 | WebFlux Gateway |
| Portal 模板 | Kotlin + Thymeleaf | SSR 降低前端复杂度，Kotlin 与 Java 混编零摩擦 | React/Vue SPA |
| 持久化 | MySQL 8.4 + Flyway | 事务完整性，版本化迁移 | MongoDB、ORM-less |
| 缓存 | Redis 7.4 | 原子操作（Lua 脚本限流、红包池）、会话 | Memcached |
| 搜索 | Meilisearch 1.12 | 零配置开箱即用，毫秒级响应，本地 Kind 友好 | Elasticsearch |
| 消息 | Apache Kafka KRaft | Outbox Pattern 天然契合，无 ZooKeeper 依赖 | RabbitMQ |
| 支付 | Stripe SDK | Webhook 回调成熟，本地测试工具链完善 | PayPal（Phase 4 扩展） |
| 特性开关 | OpenFeature | 标准化 SPI，支持运行时热更新 | LaunchDarkly（成本） |
| 弹性 | Resilience4j | Spring Boot 原生集成，注解驱动 | Hystrix（已停维护） |
| 可观测性 | Micrometer + OTLP | 三支柱统一采集，Prometheus 生态 | 专有 APM |

### 4.4 `tech-stack/best-practices.md`（整合 + 扩充）

### 4.5 `architecture/index.md`（从 architecture.md 精简）

- 三层模型图（Edge / BFF / Domain）
- 数据库独立隔离原则（每服务独立 schema）
- 核心/非核心解耦原则（同步 vs Kafka 异步，参见 event-driven.md）
- 各子页导航卡片

### 4.6 `architecture/bff-pattern.md`（从 modules/bff-portals.md 移入，重写定位）

**定位：解释 BFF 架构模式，以本项目为例**（而非服务 API 参考手册）

内容：
- 为什么需要 BFF：聚合 vs 领域服务直接暴露的权衡
- Thin BFF 原则：BFF 只做聚合，不放业务逻辑
- Buyer / Seller 分离的设计依据
- Circuit Breaker 降级策略（promotion / loyalty / search 三种降级行为）
- 与 Portal 的关系（Thymeleaf SSR 调用同一 BFF API）

> Buyer BFF、Seller BFF、Buyer Portal、Seller Portal 各自的 API 表保留在
> `docs/services/` 对应的详细设计文档中，不在此页重复。

### 4.7 `architecture/event-driven.md`（新建）

- Outbox Pattern 原理图（事务写 outbox → Poller 5s → Kafka → Consumer）
- 当前 Kafka topic 清单（`order.events.v1`、`wallet.transactions.v1`、`marketplace.product.events.v1` 等）
- Consumer 幂等规范（事件 ID 去重策略）
- 核心/非核心服务解耦：何时用同步（快速失败 + CB）、何时用事件（天然异步）

> **实现状态说明**：`activity-service` 的 `RewardDispatcher`（奖励派发到 loyalty/wallet）
> 当前为桩实现，实际积分/钱包派发通过 Kafka 事件完成，reward_status 补录机制仍在规划中。
> 文档中凡涉及奖励派发链路，均以"当前实现"与"目标态"分别标注。

### 4.8 `services/core.md`

**服务分组依据**：直接决定"能否浏览、加购、下单、支付、查单"的服务

覆盖（含 Profile Service）：

| 服务 | 端口 | 职责 |
|------|------|------|
| Auth Server | 8090 | JWT 签发、Google OAuth2、Guest Token |
| Profile Service | 8083 | 用户档案、地址簿、卖家信息 |
| Marketplace Service | 8084 | 商品目录、SKU、买家评价 |
| Order Service | 8085 | 订单状态机、Outbox 事件、游客订单 |
| Wallet Service | 8086 | 余额、充值、Stripe 支付意图 |

每服务：职责一句话、端口、核心 API 表（3-5 条）、与其他服务的关系

> **Auth Server 说明**：现有 `modules/auth-server.md`（69 行）内容不完整，
> 需以 `domain-services.md` auth-server 节（含 Google OAuth2、Guest Token、社交绑定）
> 为权威来源，并补充原创内容覆盖完整 API 表。

### 4.9 `services/growth.md`

**服务分组依据**：提升用户留存与转化，不直接阻断下单链路

覆盖：Loyalty Service（积分生命周期）、Promotion Service（双轨模型）、Notification Service（Channel SPI）

### 4.10 `services/platform.md`

**服务分组依据**：基础设施能力，为业务服务提供横向支撑

覆盖：Activity Service（GamePlugin SPI）、Search Service（Feature Toggle + BFF 直连）、Webhook Service、Subscription Service

> Search Service 归入 platform 而非 core，原因：其不在 API Gateway 路由链上，
> 由 BFF 直接调用，更接近"基础设施能力"而非"交易主链路"。
