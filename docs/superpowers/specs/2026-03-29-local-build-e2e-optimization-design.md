# Shop Platform 本地构建 / E2E 加速设计文档

> 版本：1.0 | 日期：2026-03-29

---

## 一、概述

### 1.1 目标与范围

本文档定义本地默认构建 / 部署链路的加速方案，目标是在不牺牲可验证性和可回退性的前提下，缩短开发者从“修改代码”到“在 Kind 集群中验证 buyer / seller 功能”的反馈时间。

**范围内（IN）：**

- 本地默认构建链路优先走 fast 路径
- 将 Java 服务镜像构建从 Docker 内部 Maven 编译，迁移为宿主机 Maven 增量产物复用
- 引入 `docker/Dockerfile.fast`，直接消费宿主机构建出的 JAR
- 本地镜像分发优先走 `localhost:5000` registry，而不是每次 `kind load`
- 本地部署仅针对受影响模块做镜像更新、重启与等待
- 保留旧链路作为 fallback，并在 fast 前提不满足时显式回退
- 在本地部署完成后，验证现有 buyer / seller smoke 与 UI e2e 能正常通过

**范围外（OUT）：**

- CI / GitHub Actions 流程改造
- 生产或预发环境的镜像发布策略调整
- buyer / seller 业务功能本身的变更
- 新增一套独立于现有脚本体系之外的部署平台

### 1.2 成功标准

- 本地默认链路在增量修改时优先使用 fast 路径
- 共享模块或基础设施命中时，系统能够自动扩大重建范围，而不是产生不完整部署
- 缺少宿主机构建产物、registry 不可用、模块判定失败等场景有明确 fallback
- 完成一次本地部署后，现有 buyer / seller smoke 与 UI e2e 验证通过
- 至少完成一次“增量变更 -> 快速重建 -> 本地部署 -> buyer / seller 验证”的真实演练

## 二、现状与瓶颈

当前本地链路的主要耗时来自以下几个方面：

1. `scripts/build-images.sh` 在 Docker builder stage 内重复执行 Maven，多次消耗依赖解析和模块打包时间。
2. `kind load docker-image` 在镜像较多或镜像体积较大时，拷贝成本高于本地 registry push。
3. 默认链路更偏向“全量稳定”，即使是单模块变更，也容易带出较大范围的等待和镜像装载。
4. 构建成功并不等于本地可用，仍需通过 buyer / seller smoke 与 UI e2e 确认部署后的真实可用性。

因此，优化目标不是单纯“把脚本改快”，而是把本地默认开发闭环优化为：

`变更检测 -> 宿主机增量构建 -> 轻量镜像打包 -> 快速分发 -> 选择性部署 -> buyer / seller 验证`

## 三、目标方案

### 3.1 默认本地主链路

本地默认链路调整为以下流程：

1. 检测变更模块
2. 识别是否命中共享模块 / 公共脚本，从而决定是增量重建还是扩大为全量重建
3. 在宿主机执行 Maven `package`，仅为需要打镜像的模块生成最新 JAR
4. 使用 `docker/Dockerfile.fast` 基于已有 JAR 构建镜像
5. 优先将镜像推送到 `localhost:5000`
6. 对受影响 deployment 执行选择性重启，并等待 ready / rollout 完成
7. 运行现有 smoke 与 buyer / seller UI e2e，确认功能可用

### 3.2 兼容与回退策略

尽管 fast 路径成为本地默认值，旧链路必须继续保留，作为以下场景的明确 fallback：

- 本地 registry 尚未准备好
- 目标模块 JAR 不存在或宿主机构建失败
- 模块判定脚本无法得出可信结果
- 开发者显式要求使用 legacy / full 路径

回退必须是“显式、可见、可诊断”的，不能静默跳过 fast 路径后继续执行，以免开发者误判当前运行的实际链路。

## 四、组件与文件改动

### 4.1 `docker/Dockerfile.fast`

新增一个薄镜像 Dockerfile，仅承担运行时封装职责，不再在镜像内运行 Maven。镜像构建时由脚本传入模块名与对应 JAR 路径，直接复制宿主机构建产物。

该 Dockerfile 的职责边界：

- 接收宿主机构建完成的 JAR
- 保持与当前运行时镜像一致的基础镜像与启动方式
- 不引入新的打包逻辑，不承担依赖解析职责

### 4.2 `scripts/build-images.sh`

该脚本负责成为 fast 构建的入口：

- 增加 host-build / fast 模式
- 基于变更模块集合在宿主机执行 Maven `package`
- 当命中共享模块时，自动扩大构建范围
- 在 fast 模式下切换到 `docker/Dockerfile.fast`
- 在 fast 前提不满足时，明确打印原因并回退到 legacy Docker 构建链路

### 4.3 `scripts/load-images-kind.sh` 与 registry 相关脚本

本地镜像分发优先使用 registry push。`kind load` 不会被删除，但会降级为 fallback 方案。

目标行为如下：

- 优先检测并复用 `localhost:5000`
- registry 可用时推送镜像并复用现有 tag 约定
- registry 不可用时，退回现有 Kind load 路径

### 4.4 `scripts/deploy-kind.sh`

部署脚本需要从“面向全量部署”升级为“面向受影响模块部署”：

- 继续保留现有 `kubectl apply -k` 基线
- 若是增量变更，只对受影响 deployment 做重启 / 等待
- 若命中共享模块或需要全量刷新，则保留全量等待逻辑
- 等待逻辑基于 `kubectl wait` / `kubectl rollout status`，避免固定 sleep

### 4.5 `scripts/e2e.sh` 与 `Makefile`

`make e2e` 与相关本地默认目标应优先走 fast 路径，但保留 fallback 开关：

- 默认：fast 本地构建、快速分发、选择性部署、真实验证
- fallback：显式切回 legacy 路径
- 如有需要，可以增加辅助目标（例如便于排障的 legacy 入口），但不应让默认链路继续维持在最慢路径

## 五、详细流程设计

### 5.1 变更检测与重建范围决策

变更检测以现有模块判定脚本为基础，并增加以下规则：

- 若命中 `shop-common`、`shop-contracts`、根 `pom.xml`、镜像构建公共文件等共享路径，则升级为更大范围重建
- 若仅命中单个服务模块，则仅构建该模块及其 Maven 依赖
- 若脚本无法可靠判断范围，则回退到更保守的构建范围，而不是假设“只改了一个模块”

### 5.2 宿主机构建策略

宿主机构建的核心策略为：

- 对需要出镜像的服务模块执行 Maven `package`
- 优先使用 `-pl <modules> -am` 这类增量方式复用 Maven 本地缓存
- 不引入新的构建工具或替换当前 Maven 主体流程
- 对于 seller Web UI 等现有 UI 验证所需构建，继续复用已有脚本中的构建步骤，不把 UI 逻辑混入后端镜像构建阶段

### 5.3 镜像分发与部署刷新

若镜像 tag 在本地开发环境中保持稳定，则部署刷新必须显式触发 Pod 重建，以便重新拉取最新镜像。也就是说，“推送成功”不等于“集群内已生效”，脚本需要把这一步封装完整。

### 5.4 buyer / seller 验证策略

部署后的验收分为两层：

1. **基础连通性与鉴权验证**
   - 继续运行现有 smoke，确认 gateway、buyer 接口与 seller 鉴权入口正常
2. **真实 UI / 入口验证**
   - 继续运行 buyer SSR 与 seller Web UI e2e
   - 验证本次 fast 链路不会破坏 buyer / seller 的真实入口能力

最终验收不只验证首次全量部署，还要至少覆盖一次增量部署场景。

---

## 六、错误处理与可诊断性

本次优化必须遵守以下约束：

- 不增加静默 fallback；任何回退都要打印明确原因
- 不因模块判定不准而直接跳过部署或验证
- 对缺失产物、registry 不可用、Deployment 等待失败等情况，返回清晰的失败信号
- 保持现有脚本体系可单独执行，便于定位是“构建问题”“分发问题”还是“部署 / 验证问题”

---

## 七、验证与验收计划

实现完成后，至少执行以下验证：

1. 针对受影响构建脚本运行现有仓库检查命令，确认脚本语法和调用链未被破坏
2. 完成一次真实本地构建 / 部署演练，优先验证 fast 默认链路
3. 运行现有 buyer / seller smoke 与 UI e2e，确认 buyer / seller 功能入口可正常使用
4. 至少验证一次增量变更场景，证明加速方案对本地日常开发闭环有效
5. 在 fast 前提刻意不满足的场景下验证 fallback，确认旧链路仍然可用

---

## 八、实施边界

本设计优先解决“本地默认链路仍然过慢”的问题，不追求一次性重写全部平台脚本。凡是与当前目标无直接关系的 CI、GitOps、生产流程变更，均不纳入本次实现。
